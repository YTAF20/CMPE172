1. Request Flow Through Layers
------------------------------------------------------------

The application is structured in four layers:
Controller -> Service -> Repository -> Data

When a request comes in, it goes through each layer one at a time.

Example: when a student submits the booking form (POST /appointments/book):

1. AppointmentController gets the request and pulls out the slotId
   and studentName from the form, then calls
   appointmentService.bookAppointment(slotId, studentName).

2. AppointmentService checks if the slot is still open by calling
   the repository. If it's not open it returns false and the
   controller sends the user back to the slots page. If it is open,
   it marks the slot as booked and saves a new Appointment.

3. The repositories just hold in-memory lists for now. They are
   the only place where data gets read or changed.

4. Once the booking is saved, the controller returns the
   "confirmation" view and Thymeleaf renders the confirmation page.

------------------------------------------------------------
2. Pattern: Page Controller vs Front Controller
------------------------------------------------------------

I used the Page Controller pattern for this project. This means
each page has its own controller class:

  - HomeController       -> handles GET /
  - SlotController       -> handles GET /slots and GET /slots/{id}/book
  - AppointmentController -> handles GET /appointments and POST /appointments/book

I chose Page Controller because each page does something pretty
different. The home page, the slots list, and the booking form don't
share much logic, so it made more sense to keep them separate rather
than routing everything through one big controller class.

A Front Controller puts all requests through one central class first.
That's useful if you need to do the same thing for every request like
checking if a user is logged in. For this project that wasn't needed,
so Page Controller kept things simpler and easier to follow.
