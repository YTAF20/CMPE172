Academic Advising Appointment Scheduling System

1. Request Flow Through Application Layers

This application follows a strict four-layer architecture: Controller → Service → Repository → Database. Every incoming HTTP request passes through each layer in sequence before a response is returned. No layer skips another, which keeps the system modular and testable.

Browser / Client | | HTTP Request (e.g., GET /slots or POST /appointments/book) v +----------------------+ | Controller Layer | (SlotController, AppointmentController, HomeController) | Spring @Controller | - Parses request params / path variables | | - Calls the appropriate Service method | | - Adds data to the Model for the view +----------+-----------+ | v +----------------------+ | Service Layer | (TimeSlotService, AppointmentService) | @Service | - Enforces business rules (e.g., slot must be open) | @Transactional | - Coordinates multiple Repository calls | | - Marked with @Transactional for atomic operations +----------+-----------+ | v +----------------------+ | Repository Layer | (TimeSlotRepository, AppointmentRepository, UserRepository) | @Repository | - Executes raw SQL via Spring JdbcTemplate | JdbcTemplate | - Maps ResultSet rows to model objects (RowMapper) | | - No ORM; all SQL is hand-written +----------+-----------+ | v +----------------------+ | SQLite Database | (advising.db) | | - Tables: users, advisors, time_slots, appointments | | - Schema initialized from schema.sql on startup +----------------------+ | v Thymeleaf template rendered → HTTP Response to Browser

As a concrete example, when a student submits the booking form (POST /appointments/book), the following sequence occurs:

  • AppointmentController receives the POST, extracts slotId and studId from the form body, and calls appointmentService.bookAppointment(slotId, studId).
  • AppointmentService opens a transaction (@Transactional), calls TimeSlotRepository.findById(slotId) to verify the slot is still open, then calls TimeSlotRepository.markSlotBooked() and AppointmentRepository.save() atomically.
  • TimeSlotRepository / AppointmentRepository execute the corresponding SQL UPDATE and INSERT statements via JdbcTemplate against the SQLite database file.
  • The controller receives a true result, adds a success message to the Model, and returns the "confirmation" view name. Thymeleaf renders confirmation.html and sends the response to the browser.

This layered approach ensures that the controller never touches the database directly, the service never concerns itself with HTTP request parsing, and the repository never contains business logic. Each layer has a single, well-defined responsibility.

2. Page Controller vs. Front Controller Pattern

This application uses the Page Controller pattern rather than a single Front Controller. In the Page Controller pattern, each logical page or endpoint is handled by its own dedicated controller class. In this project there are three page controllers:

  • HomeController — handles GET / (home page) and GET /health.
  • SlotController — handles GET /slots (slot list) and GET /slots/{id}/book (booking form).
  • AppointmentController — handles GET /appointments (appointment list), POST /appointments/book (submit booking), and POST /appointments/{id}/cancel (cancel).

The Page Controller pattern was chosen because each page in this system has a narrow, focused scope: browsing slots, booking a slot, and viewing appointments are three distinct user workflows with little shared pre-processing logic. Page controllers keep each feature isolated, which makes the code easier to read and modify without introducing a large centralized routing class.

A Front Controller (such as a single DispatcherController that routes all requests) would be more appropriate for systems that need to enforce uniform cross-cutting behavior across every request — for example, authentication checks, logging middleware, or request validation before dispatch. Spring Boot's own DispatcherServlet already fulfills this role at the framework level (routing all requests to the correct @Controller), so adding a custom application-level front controller here would be redundant.

In summary, the Page Controller pattern suits this project because the pages are logically independent, the application is small-to-medium in scope, and the framework's built-in dispatcher provides the centralized routing already.
